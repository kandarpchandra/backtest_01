# PART 1: THE BASICS — What Is This Project?

## What is Backtesting?

Imagine you have an idea: "I think if I buy a stock whenever its 10-day average price crosses above its 20-day average price, and sell when it crosses back down, I'll make money."

Before risking real money, you want to test this idea on historical data. You want to ask: *"If I had followed this exact rule for the last 2 years, how much money would I have made or lost?"*

That process of simulating your trading idea on past market data is called **Backtesting**.

This project is the engine that runs that simulation.

## Why Not Just Use a Spreadsheet?

A spreadsheet (or a simple Python script with Pandas) calculates everything at once. It looks at ALL the data — past, present, and future — simultaneously. This creates a deadly problem called **Look-ahead Bias**: your spreadsheet might accidentally use tomorrow's price to decide what to buy today. In real life, you can't see the future.

Our engine solves this by processing data **one tick at a time**, in strict chronological order, just like a real trading desk. Your strategy code literally cannot access data from the future, because it hasn't arrived yet.

## Why Event-Driven?

Think of a real trading floor:
1. A price update comes in on the screen.
2. The trader sees it and decides to buy.
3. The trader tells the broker how many shares.
4. The broker sends the order to the exchange.
5. The exchange fills the order and sends back a confirmation.
6. The accountant updates the books.

Each of those steps is an **Event**. Our engine models this exact flow. Each step generates an Event object, places it in a queue, and the next step picks it up and processes it. Nothing happens out of order. Nothing is skipped.

# PART 2: THE EVENT SYSTEM — The Nervous System of the Engine

## Events — The Messages

Every piece of communication in the engine is an **Event**. An Event is a small, immutable (cannot be changed after creation) data packet. It contains:
- **`timestamp`**: When this event happened (a float representing seconds).
- **`event_id`**: A unique UUID so no two events are ever confused.
- **`event_type`**: What kind of event this is (BAR, SIGNAL, ORDER, FILL, LIFECYCLE).

### File: `src/backtester/events/types.py`

This file defines every type of Event in the system:

**`BaseEvent`** — The parent class. All events inherit from this. It implements `__lt__` (less-than comparison) so events can be sorted. Sorting order:
1. First by **timestamp** (earlier events first).
2. If timestamps are equal, by **priority** (market data before signals, signals before orders...).
3. If priorities are also equal, by **event_id** (just to break ties deterministically).

**`MarketDataEvent`** — Base class for all incoming market data. Has a `symbol` field (like "AAPL").
- **`TradeBarEvent`** — Standard OHLCV bar: `open`, `high`, `low`, `close`, `volume`. This is what your strategy reads to make decisions.
- **`OptionDataEvent`** — Contains `bid`, `ask`, `iv` (implied volatility), and all Greeks (`delta`, `gamma`, `theta`, `vega`), plus `underlying_price`. Used by the Black-Scholes pricer.
- **`YieldDataEvent`** — Contains `ytm` (yield to maturity), `duration`, and `convexity`. Used by the bond pricer.

**`SignalEvent`** — Generated by your strategy. Contains:
- `symbol`: Which asset to trade.
- `direction`: `LONG` (buy), `SHORT` (sell), or `FLAT` (close position).
- `strategy_id`: Which strategy generated this signal (important for multi-strategy backtests).
- `strength`: A multiplier (default 1.0) that the allocator can use to scale position size.

**`OrderEvent`** — Generated by the Capital Allocator. Contains:
- `symbol`, `direction`, `quantity`: What to buy/sell and how much.
- `order_type`: `MARKET` (fill at current price), `LIMIT` (fill only at my price or better), or `STOP` (trigger when price reaches a level).
- `price`: Required for LIMIT and STOP orders.

**`FillEvent`** — Generated by the Execution Engine. Contains:
- `symbol`, `direction`, `quantity`: What was actually filled.
- `fill_price`: The actual price you got (after slippage).
- `commission`: How much the broker charged you.

**`LifecycleEvent`** — Generated by the system for corporate actions. Has a `symbol` field.
- **`OptionExpiryEvent`** — Contains `underlying_price`. Tells the lifecycle handler to settle or expire the option.
- **`CouponPaymentEvent`** — Contains `coupon_amount`. Tells the lifecycle handler to credit bond interest.

### The Priority System

The `EventType` enum uses **explicit integer values**, not `auto()`. This is critical:

```
BAR       = 10   (processed first)
SIGNAL    = 20
ORDER     = 30
FILL      = 40
LIFECYCLE = 50   (processed last)
```

**Why explicit values?** If someone accidentally reorders the enum definition, `auto()` would silently reassign the numbers, and the entire engine would process events in the wrong order. With explicit values, the contract is self-documenting and safe.

## The Queue — The Post Office

### File: `src/backtester/events/queue.py`

The `SyncEventQueue` wraps Python's built-in `queue.PriorityQueue`. When you `put()` an event, it's inserted in sorted order. When you `get()`, you always receive the highest-priority (lowest number) event.

Think of it as a post office that always delivers the most urgent letter first.

## The Clock — The Timekeeper

### File: `src/backtester/core/clock.py`

The `SimulationClock` has one job: **prevent time from going backwards**.

```python
def advance(self, next_time: float) -> None:
    if next_time < self._current_time:
        raise ValueError(f"Cannot move time backwards: {self._current_time} -> {next_time}")
    self._current_time = next_time
```

Every time the engine processes an event, it advances the clock to that event's timestamp. If somehow an event from the past sneaks in, the engine crashes immediately rather than silently producing wrong results.

# PART 3: THE DATA LAYER — Feeding the Engine

## Data Feeds

### File: `src/backtester/data/feed.py`

**`AbstractDataFeed`** — The interface. Any data source must implement one method:
```python
def next(self) -> Optional[TradeBarEvent]:
```
Call `next()`, and it returns the next market data event. When the data is exhausted, it returns `None`.

**`SyntheticFeed`** — A built-in data generator for testing. It creates a realistic random-walk price series using **Geometric Brownian Motion (GBM)** — the same mathematical model used in the Black-Scholes equation.

Key detail: Each `SyntheticFeed` instance uses its own private random number generator (`np.random.default_rng(seed)`). This means if you create feeds for AAPL (seed=42) and MSFT (seed=99), they generate completely independent price series. An earlier version used the global `np.random.seed()`, which meant the second feed would corrupt the first feed's random state.

The volume is clamped to `max(1, ...)` to prevent negative volumes (which would cause the execution engine to calculate negative fill quantities).

## Multi-Symbol Synchronization

### File: `src/backtester/data/sync.py`

**The Problem:** In real life, AAPL and MSFT don't print trades at the exact same millisecond. If AAPL ticks at 10:00:01.003 and MSFT ticks at 10:00:01.001, the engine must process MSFT first.

**`MultiSymbolSynchronizer`** — Takes a list of data feeds and acts as a single unified feed. Internally, it:
1. **Primes the pump**: Reads the first bar from every feed at initialization.
2. **On each `next()` call**: Looks at the head of every feed, finds the one with the earliest timestamp, returns it, and advances only that specific feed.

This guarantees strict chronological ordering across all assets. The strategy never accidentally sees AAPL's 10:01 bar before MSFT's 10:00 bar.

# PART 4: THE STRATEGY LAYER — Making Decisions

### File: `src/backtester/strategy/base.py`

**`Strategy`** — Abstract base class. You must implement:
```python
def on_bar(self, bar: TradeBarEvent, queue: AbstractEventQueue) -> None:
```

The engine calls `on_bar()` every time a new bar arrives. You analyze the bar, and if your indicators trigger, you call `self._emit(signal, queue)` to push a `SignalEvent` into the queue.

**Critical design decision:** The strategy receives the `queue` as a parameter, but it can ONLY push `SignalEvent`s into it (via `_emit`). It cannot directly modify the portfolio, place orders, or read other strategy's signals. This isolation prevents cheating.

### File: `src/backtester/strategy/warmup.py`

**`WarmupWrapper`** — A decorator pattern. It wraps your strategy and intercepts the first N bars per symbol.

**The Problem:** A 20-period moving average needs 20 bars of data before it produces a meaningful value. If your strategy starts trading on bar 1, the moving average is garbage and produces false signals.

**The Solution:** During warmup, bars are forwarded to the strategy (so it populates its internal price arrays), but any `SignalEvent` the strategy emits is captured in a dummy queue and thrown away.

**Per-symbol tracking:** With two symbols (AAPL and MSFT), bars arrive interleaved. The wrapper tracks `_bars_per_symbol = {"AAPL": 0, "MSFT": 0}` and only allows signals for AAPL once AAPL has received its full warmup, independently of MSFT.

# PART 5: THE SIZING LAYER — How Much to Buy

### File: `src/backtester/sizing/base.py`

When the strategy says "I want to go LONG on AAPL", it doesn't specify how many shares. The **Capital Allocator** converts that qualitative signal into a quantitative order.

**`FixedLotSizer`** — Always buys the same fixed number of shares (e.g., 100). Simple but doesn't adapt to account size.

**`PercentEquitySizer`** — Allocates a percentage of your total equity per trade. If you have $100,000 and `percent=0.20`, it allocates $20,000 worth of AAPL shares. The quantity is calculated as:
```
qty = round($20,000 / current_AAPL_price)
```

**Handling FLAT signals:** When the strategy says FLAT (close position), the allocator checks how many shares you currently hold and generates an order to sell exactly that amount.

### File: `src/backtester/sizing/vol_target.py`

**`VolTargetSizer`** — An institutional-grade allocator that sizes positions inversely proportional to their recent realized volatility. If a stock is very volatile, you buy fewer shares to maintain constant risk exposure. This is how most systematic hedge funds size positions.

# PART 6: THE RISK LAYER — Protecting the Portfolio

## Pre-Trade Risk (Before execution)

### File: `src/backtester/risk/pre_trade.py`

**`PositionLimitCheck`** — You configure maximum position sizes per symbol (e.g., `{"AAPL": 1000, "MSFT": 500}`). Before any order reaches the execution engine, this check calculates what the new position would be. If it exceeds the limit, the order is silently dropped.

**`PreTradeRiskEngine`** — A chain of checks. You can add as many as you want. An order must pass ALL checks to proceed.

## Post-Trade Risk (After execution)

### File: `src/backtester/risk/post_trade.py`

**`PostTradeRiskEngine`** — After every fill, evaluates the health of the portfolio.

### File: `src/backtester/risk/drawdown.py`

**`DrawdownControl`** — Tracks the peak equity (highest account value ever reached). After every fill, it calculates:
```
drawdown = (peak - current) / peak
```
If this exceeds your threshold (e.g., 20%), trading is permanently halted by returning `FLATTEN_ALL`, which causes the engine to liquidate all positions.

### File: `src/backtester/risk/budget.py`

**`DailyLossLimit`** — Tracks how much you've lost today. If intraday losses exceed a threshold, trading stops for the day.

# PART 7: THE EXECUTION LAYER — Simulating the Exchange

This is where the engine gets truly institutional-grade.

## The Order Management System (OMS)

### File: `src/backtester/execution/oms.py`

**`OrderState`** — A mutable wrapper around the immutable `OrderEvent`. It tracks:
- `status`: `PENDING` → `ACCEPTED` → `PARTIAL` → `FILLED`
- `filled_quantity`: How much has been filled so far.

**Why a wrapper?** The `OrderEvent` is frozen (immutable) to guarantee data integrity in the event queue. But an order's status needs to change over time. The `OrderState` wrapper solves this by being a mutable container around the immutable event.

**`OrderTracker`** — The OMS. It maintains two dictionaries:
- `active_orders`: Orders that are still working (waiting for fills).
- `closed_orders`: Orders that have been fully filled or cancelled.

**`close_order()`** — Moves a filled order from active to closed **without changing its status**. An earlier version incorrectly used `cancel_order()` for this, which overwrote the `FILLED` status to `CANCELLED` — meaning every completed trade appeared as cancelled in the trade history.

## The Execution Engine

### File: `src/backtester/execution/engine.py`

**`OrderBookMatcher`** — The heart of execution. On every new bar, it:

1. **Updates its internal market data** (`current_bars[symbol] = bar`).
2. **Scans all active orders** for that symbol.
3. **Evaluates each order** against the bar:
   - **MARKET orders**: Fill at the bar's close price.
   - **LIMIT orders**: For a buy, fill only if the bar's low reached the limit price. For a sell, fill only if the bar's high reached the limit price.
   - **STOP orders**: For a buy-stop, trigger only if the bar's high reached the stop price. For a sell-stop, trigger only if the bar's low reached the stop price.

4. **Simulates partial fills**: You can only fill up to 10% of the bar's traded volume. If you want to buy 5,000 shares but only 1,000 traded, you get filled for 100 shares (10% of 1,000). The rest stays as a PARTIAL order and tries again on the next bar.

5. **Applies slippage**: The fill price is adjusted adversely based on your market participation rate.

6. **Applies transaction costs**: Commission is calculated and included in the `FillEvent`.

## Slippage Models

### File: `src/backtester/execution/slippage.py`

**`ZeroSlippage`** — No slippage (useful for testing).

**`FixedBasisPointSlippage`** — A flat percentage penalty. For example, 1 basis point = 0.01% of the fill price, always adverse (you pay more when buying, receive less when selling).

**`VolumeLinearSlippage`** — The institutional-grade model. Slippage is proportional to your **participation rate**:
```
participation_rate = order_quantity / bar_volume
slippage = fill_price × participation_rate × impact_factor
```
If you're trying to buy 10% of the day's volume, you'll move the market significantly. If you're only 0.1%, the impact is negligible.

## Transaction Cost Models

### File: `src/backtester/execution/transaction_cost.py`

**`ZeroTCM`** — No commissions.
**`PerShareTCM`** — Fixed cost per share (e.g., $0.005/share, like Interactive Brokers).
**`PercentOfValueTCM`** — Cost as a percentage of trade value (e.g., 5 basis points of notional).

# PART 8: THE PORTFOLIO LAYER — Tracking Your Money

## The Account Model

### File: `src/backtester/portfolio/pnl.py`

**`AccountModel`** — The most critical module. It is the single source of truth for your financial state.

**State tracked:**
- `available_cash`: How much cash you have right now.
- `total_equity`: Cash + Market Value of all positions.
- `positions`: Dictionary of `{symbol: quantity}`.
- `cost_basis`: Dictionary of `{symbol: average_entry_price}`.
- `realized_pnl`: Total profit/loss from closed trades.
- `total_commission`: Total fees paid.
- `equity_curve`: A list of equity snapshots over time (used for Sharpe ratio, drawdown, etc.).

### How `on_fill()` Works — The Core Accounting Logic

When a `FillEvent` arrives, the following happens:

**Step 1: Deduct Commission**
```python
self.total_commission += fill.commission
self.available_cash -= fill.commission
```

**Step 2: Check for Decomposable Instruments**
If the instrument is a `STIRSpread` or `STIRFly` (implements `Decomposable`), the fill is NOT booked against the synthetic ticker. Instead:
- The spread price is deducted from cash.
- `decompose(qty)` is called, which returns `{"ERZ4": 1.0, "ERH5": -1.0}` (for a spread).
- Each leg is booked individually into the `positions` dictionary.
- This allows cross-margining and legging out.

**Step 3: Update Position (`_update_position`)**
Three scenarios:
1. **Opening a new position** (current quantity is 0): Simply set position and cost basis.
2. **Adding to an existing position** (same direction): Calculate the **Volume Weighted Average Price (VWAP)** of the combined position:
   ```
   new_cost_basis = (old_cb × old_qty + fill_price × fill_qty) / total_qty
   ```
3. **Closing or flipping** (opposite direction): 
   - Calculate realized PnL: `(fill_price - cost_basis) × closed_quantity` for longs.
   - If the fill quantity exceeds the current position, the position **flips** (e.g., from Long 2 to Short 1). The new cost basis becomes the fill price.

**Step 4: Recalculate Account**
```python
portfolio_value = available_cash + sum(qty × current_price for each position)
```

## Margin Tracking

### File: `src/backtester/portfolio/margin.py`

**`MarginModel`** — Abstract interface for calculating margin requirements.
**`EquityMarginModel`** — Standard Reg-T margin for equities (50% initial, 25% maintenance).

The `AccountModel` recalculates margin requirements on every tick. If `available_cash` drops below `maintenance_margin_req`, the portfolio is at risk of a margin call.

## The Lifecycle Handler — Corporate Actions

### File: `src/backtester/portfolio/lifecycle.py`

**`LifecycleHandler`** — Handles non-trade events automatically.

**Option Expiry:**
1. Check if we hold the option.
2. Calculate intrinsic value: `max(0, underlying_price - strike)` for calls.
3. Multiply by position size and the 100x contract multiplier.
4. If we're long, credit cash. If we're short, debit cash.
5. Remove the position and cost basis.

**Coupon Payment:**
1. Calculate payment: `coupon_amount × position_size`.
2. Credit to cash.

# PART 9: THE PRICING LAYER — Valuing Non-Linear Assets

## The Problem

For an Equity, if the price goes from $100 to $101, you made $1 per share. Simple.

For an Option, if the underlying stock goes from $100 to $101, you did NOT make $1. You made some non-linear amount that depends on the strike price, time to expiration, implied volatility, and risk-free rate. The relationship is governed by the **Black-Scholes equation**.

## The Solution: Pluggable Pricers

### File: `src/backtester/pricing/base.py`

**`AbstractPricer`** — Interface with one method: `get_value(data: MarketDataEvent) -> float`.

**`LinearPricer`** — For equities and futures: `value = price × multiplier`. An AAPL share has multiplier 1.0. A Euribor future has multiplier `tick_value / tick_size = 12.50 / 0.005 = 2500`.

### File: `src/backtester/pricing/options.py`

**`BlackScholesPricer`** — Implements the Black-Scholes formula for European options:

```
d1 = (ln(S/K) + (r + σ²/2)T) / (σ√T)
d2 = d1 - σ√T

Call Price = S·N(d1) - K·e^(-rT)·N(d2)
Put Price  = K·e^(-rT)·N(-d2) - S·N(-d1)
```

Where:
- `S` = underlying stock price (from `OptionDataEvent.underlying_price`)
- `K` = strike price
- `T` = time to expiration in years
- `r` = risk-free interest rate
- `σ` = implied volatility (from `OptionDataEvent.iv`)
- `N()` = cumulative normal distribution (from `scipy.stats.norm.cdf`)

At expiration (T=0), it returns the intrinsic value: `max(0, S-K)` for calls, `max(0, K-S)` for puts.

### File: `src/backtester/pricing/bonds.py`

**`YieldToPricePricer`** — Approximates bond price changes using duration and convexity:
```
ΔPrice ≈ -Duration × ΔYTM + 0.5 × Convexity × (ΔYTM)²
```

## How Instruments Use Pricers

### File: `src/backtester/instruments/base.py`

Every `Instrument` holds a reference to a `Pricer`:
```python
class Instrument(ABC):
    def __init__(self, symbol: str, pricer: AbstractPricer):
        self.symbol = symbol
        self.pricer = pricer
```

- `Equity("AAPL")` uses `LinearPricer(multiplier=1.0)`.
- `STIROutright("ERZ4", tick_size=0.005, tick_value=12.50)` uses `LinearPricer(multiplier=2500)`.
- `VanillaOption("AAPL_CALL_150", strike=150, expiry_years=0.5, is_call=True)` uses `BlackScholesPricer`.
- `FixedRateBond("UST_10Y", face_value=1000, coupon_rate=0.05)` uses `YieldToPricePricer`.

# PART 10: SYNTHETIC INSTRUMENTS & LEG DECOMPOSITION

## The Problem

In STIR futures trading, you can trade directly on the exchange in:
- **Outrights**: Buy 1 ERZ4 contract.
- **Spreads**: Buy 1 "ERZ4-ERH5" calendar spread (simultaneously buy Z4 and sell H5).
- **Butterflies**: Buy 1 "ERZ4-ERH5-ERM5" fly (buy Z4, sell 2×H5, buy M5).

Real trading scenario: You buy a spread. It goes against you. You hedge by buying the other half, which creates a butterfly. Later, you dump one leg and let the other run. 

If the portfolio tracks these as isolated instruments ("1 Spread" and "1 Outright"), you can never net them against each other. The clearinghouse wouldn't let you exit the spread by selling an outright.

## The Solution: The `Decomposable` Interface

### File: `src/backtester/instruments/base.py`

```python
class Decomposable(ABC):
    @abstractmethod
    def decompose(self, quantity: float) -> dict[str, float]:
        pass
```

**`STIRSpread`** implements:
```python
def decompose(self, quantity):
    return {"ERZ4": quantity, "ERH5": -quantity}
```

**`STIRFly`** implements:
```python
def decompose(self, quantity):
    return {"ERZ4": quantity, "ERH5": -2.0 * quantity, "ERM5": quantity}
```

## How It Works in the Portfolio

When `AccountModel.on_fill()` receives a fill for a `Decomposable` instrument:
1. It deducts the synthetic cash amount (spread_price × quantity).
2. It calls `decompose(quantity)` to get the individual leg positions.
3. It books each leg separately into `self.positions`.

**Result:** Your portfolio holds `{"ERZ4": 1, "ERH5": -1}` instead of `{"ERZ4-ERH5": 1}`. You can now sell `ERZ4` outright and realize PnL on just that leg.

# PART 11: THE ENGINE LOOP — Bringing It All Together

### File: `src/backtester/core/engine.py`

The `BacktestEngine.run()` method is the heartbeat:

```python
while True:
    if self.queue.empty():
        bar = self.data_feed.next()
        if bar is None:
            break  # Data exhausted, backtest over
        self.queue.put(bar)
    
    event = self.queue.get()
    self._process_event(event)
```

The `_process_event()` router:
1. **Log the event** to the `EventSerializer`.
2. **Advance the clock** to the event's timestamp.
3. **Route by type:**
   - `BAR` → Update execution engine prices, update portfolio market price, call `strategy.on_bar()`.
   - `SIGNAL` → Call `allocator.allocate()` to create an `OrderEvent`, run pre-trade risk checks, put order in queue.
   - `ORDER` → Call `execution.on_order()` which evaluates fills and may put `FillEvent`s in queue.
   - `FILL` → Call `portfolio.on_fill()` to update positions and cash. Run post-trade risk checks.
   - `LIFECYCLE` → Call `lifecycle_handler.process_lifecycle_event()`.

# PART 12: ANALYTICS & REPORTING

### File: `src/backtester/analytics/metrics.py`

Pure mathematical functions (no side effects):

- **`calculate_returns(equity_curve)`**: Converts an equity curve into a percentage-return series using `np.diff(prices) / prices[:-1]`.
- **`sharpe_ratio(returns)`**: `(mean_return - risk_free_rate) / std_deviation × √252`. The √252 annualizes the ratio (252 trading days/year).
- **`sortino_ratio(returns)`**: Like Sharpe, but only penalizes **downside** volatility. Uses only negative returns for the denominator. More meaningful for strategies that have asymmetric return profiles.
- **`max_drawdown(equity_curve)`**: Tracks the running peak and calculates the maximum percentage drop from peak to trough.

### File: `src/backtester/reporting/tearsheet.py`

**`Tearsheet`** — Generates a human-readable performance report:
- `print_stats()`: Outputs Final Equity, Realized PnL, Total Commission, Sharpe, Sortino, and Max Drawdown to the console.
- `plot()`: Creates an interactive HTML chart of the equity curve using Plotly.

### File: `src/backtester/core/recorder.py`

**`EventSerializer`** — Records every single event during the backtest. At the end, saves them as a compressed Parquet file. This allows you to:
1. **Replay** the exact sequence of events for debugging.
2. **Analyze** individual strategy performance by filtering events by `strategy_id`.
3. **Audit** every decision the engine made.

# PART 14: THE COMPLETE STORY — How a Backtest Runs (In Layman's Terms)

Imagine the backtesting engine as a **theatre play** with several actors, each with a specific role. The play unfolds one moment at a time, in strict order, and no actor is allowed to peek at the script to see what happens next.

## The Cast

- **The Data Feed** — The narrator. It reads the script (historical price data) one line at a time.
- **The Synchronizer** — The stage manager. When multiple narrators are reading (AAPL and MSFT), the stage manager decides who speaks next based on whose line comes first chronologically.
- **The Event Queue** — The mailbox. Every actor communicates by dropping a letter (Event) into this mailbox. The mailbox always delivers the most urgent letter first.
- **The Clock** — The timekeeper. It only moves forward. If anyone tries to deliver a letter from the past, the entire play is stopped immediately (this is how we prevent cheating).
- **The Strategy** — The trader. It listens to the narrator, looks at its charts and indicators, and when it sees a pattern it likes, it writes a letter saying "I want to buy AAPL!" and drops it in the mailbox.
- **The Allocator** — The risk manager. It picks up the trader's letter and decides exactly how many shares to buy based on the account size. It writes a new, more specific letter: "Buy 133 shares of AAPL at market."
- **The Pre-Trade Risk Check** — The compliance officer. Before the letter reaches the exchange, the compliance officer checks: "Does this order put us over our position limit?" If yes, the letter is shredded.
- **The Execution Engine** — The exchange. It receives the order and tries to fill it. But it's realistic: if there weren't enough sellers in the market (low volume), it can only fill part of the order. It also makes you pay a slightly worse price (slippage) and charges you a commission. It writes back: "We filled 100 of your 133 shares at $150.02, commission $0.75."
- **The Portfolio** — The accountant. It receives the fill confirmation and updates the books: subtracts cash, adds shares, records the cost basis. Then it recalculates total equity.
- **The Post-Trade Risk** — The risk monitor. After the accountant updates the books, the monitor checks: "Have we lost more than 20% from our peak? If so, FLATTEN EVERYTHING."
- **The Lifecycle Handler** — The corporate actions desk. If an option expires or a bond pays a coupon, this desk automatically settles the cash without anyone placing a trade.

## Act 1: Setup (Before the Play Starts)

Before the curtain rises, we configure everyone:

```python
config = BacktestConfig(initial_cash=100000.0)
engine = BacktestEngine(config)

# Create the actors
feed_aapl = SyntheticFeed("AAPL", start_price=150.0, seed=42)
feed_msft = SyntheticFeed("MSFT", start_price=300.0, seed=99)
feed = MultiSymbolSynchronizer([feed_aapl, feed_msft])

strategy = WarmupWrapper(MovingAverageCrossover(), warmup_bars=25)
portfolio = AccountModel(initial_cash=100000.0, instrument_registry={"AAPL": Equity("AAPL"), "MSFT": Equity("MSFT")})

# Wire them together
engine.set_feed(feed)
engine.set_strategy(strategy)
engine.set_portfolio(portfolio)
# ... etc
```

## Act 2: The Play Begins

**Scene 1 — The first tick arrives.**
The Data Feed reads: "At time 1.0, AAPL opened at 150.30, hit a high of 150.75, low of 149.80, and closed at 150.10 on 10,234 shares of volume."

The Synchronizer checks: AAPL's next bar is at time 1.0, MSFT's next bar is also at time 1.0. They're tied, so AAPL goes first (it was primed first).

A `TradeBarEvent` is placed in the mailbox.

**Scene 2 — The engine processes it.**
The engine picks up the letter. The recorder logs it. The clock advances to 1.0.

The Execution Engine updates its internal price book for AAPL. The Portfolio updates AAPL's market price to 150.10.

The Strategy receives the bar. It adds 150.10 to its price history. It only has 1 data point — far short of the 25 needed for warmup. The WarmupWrapper intercepts any signal and throws it away. Nothing happens.

**Scene 3 — Bars 2 through 25.**
The same thing repeats. Bars arrive, the strategy accumulates prices, but warmup is active. No trades are placed. The indicators are quietly hydrating.

**Scene 4 — Bar 26. First real signal.**
The strategy now has 26 AAPL prices. It calculates:
- Short MA (10-period average) = 151.20
- Long MA (20-period average) = 150.80

Short MA > Long MA! The strategy writes a letter: `SignalEvent(symbol="AAPL", direction=LONG)` and drops it in the mailbox.

**Scene 5 — The allocator sizes it.**
The engine picks up the SignalEvent. It calls the PercentEquitySizer:
```
equity = $100,000
allocation = 20% = $20,000
AAPL price = $151.20
quantity = round($20,000 / $151.20) = 132 shares
```

It creates `OrderEvent(symbol="AAPL", direction=LONG, quantity=132, order_type=MARKET)`.

The PreTradeRiskEngine checks: "Is 132 shares under our 1,000 share limit for AAPL?" Yes. The order goes into the mailbox.

**Scene 6 — The execution engine fills it.**
The OrderBookMatcher picks up the order. The latest AAPL bar had volume of 10,234.
- Maximum fill: 10% of 10,234 = 1,023 shares. Our 132-share order fits easily.
- Base price: $151.20 (bar close).
- Slippage: participation_rate = 132/10,234 = 0.013. Impact = $151.20 × 0.013 × 0.01 = $0.02. So actual fill price = $151.22.
- Commission: 132 × $151.22 × 0.0005 = $9.98.

A `FillEvent` is emitted: 132 shares of AAPL at $151.22, commission $9.98.

**Scene 7 — The portfolio books it.**
The AccountModel receives the fill:
- `available_cash = $100,000 - $9.98 - (132 × $151.22) = $79,928.86`
- `positions = {"AAPL": 132}`
- `cost_basis = {"AAPL": 151.22}`

The PostTradeRiskEngine checks drawdown: peak was $100,000, current equity is ~$99,990 (slight slippage loss). Drawdown = 0.01%. Well under 20%. Continue trading.

**Scene 8 — Many bars later, the crossover reverses.**
Short MA drops below Long MA. The strategy emits `SignalEvent(direction=FLAT)`.

The allocator sees FLAT, checks the portfolio: we hold 132 AAPL shares. It creates an order to sell 132 shares.

The execution engine fills at $152.50 (the market moved up). Commission $10.06.

The portfolio books it:
- Realized PnL = (152.50 - 151.22) × 132 = $168.96
- Cash is restored plus the profit.

**Scene 9 — The data runs out.**
After 200 bars for each symbol, the Synchronizer returns `None`. The engine's while loop breaks. The backtest is over.

**Scene 10 — The report.**
The EventSerializer saves 479 events to `event_log.parquet`.
The Tearsheet calculates:
- Final Equity: $99,130.19
- Sharpe Ratio: -0.1052
- Max Drawdown: 5.46%

The equity curve is plotted and saved as `equity_curve.html`.

**The curtain falls.**

# PART 15: INTERVIEW Q&A CHEAT SHEET

**Q: "How do you prevent look-ahead bias?"**
> "The SimulationClock enforces strict monotonic time. The strategy receives data one bar at a time via the event queue. Future data physically doesn't exist in the system yet — the data feed hasn't yielded it. If any component tries to process an event from the past, the clock raises a fatal ValueError."

**Q: "How do you handle partial fills?"**
> "The OrderBookMatcher limits fills to 10% of the bar's traded volume. If your order is larger, only a fraction fills, the OrderState is set to PARTIAL, and the remaining quantity tries again on the next bar. This prevents the backtester from assuming infinite liquidity."

**Q: "What happens if a strategy buys a Butterfly spread and then sells one leg?"**
> "The portfolio acts as a clearinghouse. When a Butterfly fill arrives, the AccountModel checks if the instrument is Decomposable. If so, it calls decompose() to shatter the spread into its three outright futures legs (+1 front, -2 mid, +1 back) and books them individually. From that point, each leg can be traded independently."

**Q: "How do you value Options?"**
> "Each instrument holds a reference to an AbstractPricer. For Options, this is a BlackScholesPricer that uses scipy.stats.norm.cdf to calculate the cumulative normal distribution. It receives OptionDataEvents containing implied volatility and the underlying price, and returns the theoretical mark-to-market value."

**Q: "If the strategy fails in live trading, how do you debug it?"**
> "The EventSerializer records every single event (ticks, signals, orders, fills) to a Parquet file. I can load the file, filter by strategy_id, and replay the exact sequence of decisions the strategy made, including the exact market state at the time of each decision."

**Q: "How does your engine handle multiple assets with different tick frequencies?"**
> "The MultiSymbolSynchronizer takes multiple independent data feeds. On each call to next(), it peeks at the head of every feed and returns the event with the earliest timestamp, advancing only that specific feed. This ensures perfect chronological ordering even with sparse, asynchronous data."

**Q: "Why did you use explicit integer values for EventType instead of auto()?"**
> "auto() assigns values based on definition order, which means reordering the enum or adding a new type silently breaks the priority system. Explicit values (BAR=10, SIGNAL=20, etc.) make the contract self-documenting. The gaps between values (10, 20, 30...) also leave room to insert new event types in between without changing existing priorities."
